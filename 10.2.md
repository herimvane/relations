# NebulaGraph 1.0.0 最终落地方案

## 目标

将当前项目从前端关系图 MVP 升级为支持 10 万级数据的关系智能分析系统。

核心原则：

- 后端存全量数据；
- 服务端完成重要度、社区、关键路径分析；
- 前端只展示局部、骨架、社区视图；
- 不再把全量图直接丢给前端；
- Excel/CSV 只支持小中规模实时分析，大规模数据必须进入数据库和异步分析流程。

如果进入实施开发，版本号定位为：

```text
1.0.0
```

---

## 一、总体架构

```text
PostgreSQL / 可选 Neo4j
        ↓
Graph Analysis Service
        ↓
Importance / Community / Backbone / Path Cache
        ↓
View API
        ↓
Frontend Local Rendering
```

前端不再请求完整图谱，而是根据视图层级请求后端返回的局部图：

```text
L0 Universe  -> 社区级视图
L1 Galaxy    -> 社区骨架视图
L2 Backbone  -> 节点核心骨架视图
L3 Local     -> 节点局部关系视图
```

## 二、重要度算法

原始公式：

```text
NodeImportance = 0.45 * PageRank + 0.35 * Betweenness + 0.20 * Degree
```

该公式适合作为初版思路，但不建议直接用于 10 万级生产数据，原因：

- 精确 Betweenness 在 10 万级图上计算成本过高；
- Degree 容易让关系很多但实际不重要的节点得分过高；
- PageRank 在大社区中容易偏向社区规模；
- 不同业务场景中“重要”的含义不同，不能只依赖拓扑指标。

### 2.1 节点重要度

建议升级为：

```text
NodeImportance =
0.35 * PageRank +
0.25 * ApproxBetweenness +
0.15 * WeightedDegree +
0.15 * CommunityCentrality +
0.10 * BusinessWeight
```

所有指标统一归一化到 `0~1`。

字段解释：

| 指标 | 含义 |
|---|---|
| PageRank | 节点被重要节点连接的程度 |
| ApproxBetweenness | 节点作为桥梁的程度，使用近似算法 |
| WeightedDegree | 加权度数，不使用简单关系数量 |
| CommunityCentrality | 节点在所属社区内的中心程度 |
| BusinessWeight | 业务权重，例如风险等级、身份、企业规模、控制比例等 |

### 2.2 边重要度

建议公式：

```text
EdgeImportance =
0.35 * EdgeBetweenness +
0.30 * Weight +
0.20 * NodeImportanceAverage +
0.15 * RelationTypePriority
```

其中：

```text
NodeImportanceAverage = (sourceImportance + targetImportance) / 2
```

字段解释：

| 指标 | 含义 |
|---|---|
| EdgeBetweenness | 边在关键连接中的桥梁程度 |
| Weight | 原始关系强度 |
| NodeImportanceAverage | 两端节点平均重要度 |
| RelationTypePriority | 关系类型优先级 |

关系类型优先级需要支持配置，例如：

```text
控制 > 持股 > 投资 > 担保 > 管理 > 亲属 > 普通关联
```

不同业务场景可以使用不同权重模板。

### 2.3 关键路径评分

关键路径不是最短路径，而是连接核心节点的高价值路径。

建议公式：

```text
PathScore =
0.45 * AvgNodeImportance +
0.40 * AvgEdgeImportance +
0.10 * PathDiversity +
0.05 * RelationTypeContinuity
- LengthPenalty
```

其中：

```text
LengthPenalty = min(0.25, PathLength * 0.03)
```

字段解释：

| 指标 | 含义 |
|---|---|
| AvgNodeImportance | 路径节点平均重要度 |
| AvgEdgeImportance | 路径边平均重要度 |
| PathDiversity | 路径是否跨社区、跨类型、跨层级 |
| RelationTypeContinuity | 关系类型是否形成有意义链条 |
| LengthPenalty | 防止过长路径获得过高分 |

## 三、10 万级数据计算策略

重要度计算不应在前端实时完成。

建议：

- 数据导入后异步计算；
- 计算结果写入数据库；
- 前端请求视图时只读取分析结果；
- 分析任务需要状态管理；
- 支持增量重算和全量重算。

### 3.1 数据规模策略

| 数据规模 | 处理方式 |
|---|---|
| <= 5,000 节点 | 可实时完整计算，可直接前端预览 |
| 5,000 ~ 20,000 节点 | 可实时 PageRank + Degree，Betweenness 使用近似 |
| 20,000 ~ 100,000 节点 | 必须异步分析，结果入库 |
| > 100,000 节点 | 分社区、分批、缓存、增量计算 |

### 3.2 Excel/CSV 限制

Excel/CSV 数据不适合作为 10 万级实时分析入口。

建议限制：

```text
前端直接预览：
<= 5,000 节点
<= 20,000 边

后端实时分析：
<= 20,000 节点
<= 80,000 边

超过限制：
进入数据库导入流程，异步计算
```

## 四、四级视图设计

### 4.1 L0 宇宙视图 Universe

目标：

展示全量图的社区结构，而不是节点明细。

后端计算：

- 使用 Louvain 或 Leiden 做社区发现；
- 每个社区生成一个聚合节点；
- 社区之间的边表示跨社区关系强度。

前端展示：

- 每个社区显示为一个星系；
- 社区大小代表节点数量；
- 亮度代表社区重要度；
- 颜色代表主要类型或风险等级；
- 点击社区进入 L1。

返回数据：

```text
community_id
community_name
node_count
edge_count
importance
risk_summary
top_node_ids
```

### 4.2 L1 星系视图 Galaxy

目标：

查看某个社区内部的骨架结构。

后端计算：

- 根据 `NodeImportance` 取社区内 Top 1000 节点；
- 找连接核心节点的关键路径；
- 返回社区内部骨架图。

前端展示：

- 显示社区核心节点；
- 显示关键路径；
- 弱化非关键节点；
- 支持点击节点进入 L2。

### 4.3 L2 骨架视图 Backbone

目标：

查看某个节点相关的核心网络骨架。

后端计算：

- 以目标节点为中心；
- 根据 `NodeImportance` 取相关 Top 1000 节点；
- 计算连接核心节点的关键路径；
- 聚合低价值外围节点。

前端展示：

- 当前节点突出显示；
- 关键路径突出显示；
- 一度和二度节点分层展示；
- 点击外围节点进入 L3 或继续下钻。

### 4.4 L3 局部视图 Local

目标：

查看具体节点的局部关系。

后端逻辑：

- 默认返回一度关系；
- 如果结果节点数小于 1000，可以完整显示；
- 如果超过 1000，只显示高重要度关系，并聚合其余节点；
- 用户继续点击节点时继续下钻。

前端展示：

- 显示当前节点局部邻域；
- 支持展开、收起；
- 支持返回上一级视图。

## 五、前端交互改造

四级视图不建议实现为四个独立页面，而应实现为同一个图谱画布上的四种数据分辨率。

核心原则：

```text
一个图谱画布
一套 GraphViewState
四种数据分辨率
带历史栈的下钻/返回
带缓存的视图请求
带相机状态恢复的平滑切换
```

前端不负责计算 L0/L1/L2/L3，只负责：

```text
记录当前视图状态
请求后端视图 API
渲染后端返回的局部图
管理下钻历史
恢复相机和选中状态
```

### 5.1 新增视图状态机

前端需要维护当前视图状态：

```ts
type ViewLevel = 'L0' | 'L1' | 'L2' | 'L3';

type GraphViewState = {
  level: ViewLevel;
  id?: string;
  title: string;
  params?: Record<string, unknown>;
  selectedNodeId?: string;
  selectedCommunityId?: string;
  filters?: Record<string, unknown>;
  camera?: {
    position: [number, number, number];
    rotation: [number, number, number];
    zoom: number;
  };
  history: GraphViewState[];
};
```

推荐视图跳转规则：

```text
L0 点击社区       -> L1 Galaxy
L1 点击核心节点   -> L2 Backbone
L2 点击节点       -> L3 Local
L3 点击可展开节点 -> L3 deeper
返回按钮          -> 恢复上一个 GraphViewState
```

### 5.2 新增返回按钮

页面增加返回按钮：

- 从 L3 返回 L2；
- 从 L2 返回 L1；
- 从 L1 返回 L0；
- L0 时禁用返回。

返回时不只是切换数据，还要恢复：

- 视图层级；
- 当前选中节点；
- 筛选条件；
- 相机位置；
- 面包屑；
- 当前数据快照缓存。

### 5.3 新增面包屑

示例：

```text
Universe / 社区 A / 贾府骨架 / 贾宝玉局部关系
```

### 5.4 视图层级视觉区分

每一层需要有明确但克制的视觉差异：

| 视图 | 视觉重点 |
|---|---|
| L0 Universe | 社区星系，大节点、少边、强调群落 |
| L1 Galaxy | 社区骨架，核心节点和关键路径 |
| L2 Backbone | 当前节点强突出，主干路径明显 |
| L3 Local | 局部邻域，节点和边标签更完整 |

界面上增加小型层级指示器：

```text
Universe / Galaxy / Backbone / Local
```

该指示器应为状态栏式信息，不建议做成大型 Tab。

### 5.5 视图切换动画

视图切换不建议直接清空画布再加载。

推荐流程：

```text
旧图淡出到 30%
新图节点从目标区域扩散出来
相机平滑移动
完成后移除旧图对象
```

动画时长建议：

```text
200ms ~ 300ms
```

如果数据较大，使用骨架 loading：

```text
正在构建 Galaxy 视图...
```

避免大面积炫光 loading。

### 5.6 视图缓存

前端应缓存最近访问过的视图：

```ts
Map<string, GraphViewResponse>
```

缓存 key 示例：

```text
L0
L1:{community_id}
L2:{node_id}
L3:{node_id}:{depth}
```

缓存上限建议：

```text
最近 20 个视图
```

缓存内容包括：

- 图谱数据；
- 当前选中节点；
- 相机状态；
- 筛选状态；
- 统计信息。

### 5.7 删除二维表抽取功能

按照需求，删除前端二维表抽取入口。

建议：

- 第一阶段先删除页面入口；
- 后端接口可以暂时保留但不暴露；
- 后续确认无用后再删除 API 和服务代码。

## 六、后端接口设计

### 6.1 分析任务

```text
POST /api/analysis/recompute
GET  /api/analysis/status/{task_id}
```

用途：

- 触发 PageRank、近似 Betweenness、社区发现、关键路径预计算；
- 查询分析任务状态。

### 6.2 视图接口

```text
GET /api/views/universe
GET /api/views/galaxy/{community_id}
GET /api/views/backbone/{node_id}
GET /api/views/local/{node_id}
```

### 6.3 关键路径接口

```text
GET /api/paths/critical?source={source_id}&target={target_id}
```

### 6.4 统一返回结构

```ts
type GraphViewResponse = {
  view_level: 'L0' | 'L1' | 'L2' | 'L3';
  title: string;
  nodes: GraphNode[];
  edges: GraphEdge[];
  communities?: CommunitySummary[];
  stats: {
    total_nodes: number;
    total_edges: number;
    visible_nodes: number;
    visible_edges: number;
    hidden_nodes: number;
  };
};
```

## 九、风险与注意事项

### 9.1 Betweenness 不可精确全量计算

10 万级数据不建议做精确 Betweenness。

建议使用：

- 采样近似；
- 社区内近似；
- 只对核心节点计算；
- 异步任务计算。

### 9.2 前端不能承担全量图

前端只负责渲染后端返回的局部图。

建议单次返回上限：

```text
节点 <= 1000
边 <= 5000
```

### 9.3 业务权重必须可配置

不同业务场景下，关系优先级不同。

需要支持配置：

- 关系类型权重；
- 节点类型权重；
- 风险等级权重；
- 业务字段权重。

## 十、最终结论

该方案可行。

但重要度算法不应只使用单一拓扑公式，而应升级为：

```text
拓扑重要度 + 社区重要度 + 业务重要度 + 关系类型权重
```

最终系统应按照如下方式运行：

```text
数据库保存全量数据
后端异步分析
结果写入数据库
前端请求分层视图
用户逐层下钻
关键路径由后端计算并解释
```

这样才能支撑 10 万级数据，同时保持前端可读、可控、可交互。
